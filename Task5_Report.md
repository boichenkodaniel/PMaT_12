# Task 5 — Отчет по модулю расчета штрафов

## 1. Как работает текущая система штрафов (для стажера)

Представь, что читатель взял книгу в библиотеке и не вернул ее вовремя. Наша система считает, сколько он должен заплатить за просрочку.

### На что опирается расчет?

У нас есть **два входных параметра**:

1. **Количество дней просрочки** — сколько дней книга просрочена.
2. **Тип книги** — художественная (`fiction`), научная (`science`) или другая (`other`).

### Из чего состоит штраф?

Штраф складывается из двух частей:

| Компонент | Что это |
|-----------|---------|
| **Базовая сумма** | Количество дней × дневной тариф |
| **Доплата (штраф)** | Фиксированная сумма, которая добавляется при превышении порогов |

### Пороги просрочки

Система смотрит на количество дней и применяет два порога:

- **7 дней** — если просрочка больше недели, добавляется **недельная доплата**.
- **30 дней** — если просрочка больше месяца, добавляется **месячная доплата**.

> Важно: доплаты — это **разовые** фиксированные суммы, а не умножение. Например, за 45 дней просрочки читатель платит: базовая сумма за 45 дней + недельная доплата + месячная доплата.

### Дневные тарифы и доплаты по типам книг

| Тип книги | Дневной тариф | Доплата за неделю | Доплата за месяц |
|-----------|--------------|-------------------|------------------|
| Художественная | 10 руб./день | 100 руб. | 500 руб. |
| Научная | 50 руб./день | 200 руб. | 1000 руб. |
| Другая | 5 руб./день | 50 руб. | 0 руб. |

### Пример расчета

Читатель просрочил **научную книгу на 10 дней**:
- Базовая сумма: 10 × 50 = 500 руб.
- Просрочка больше 7 дней → доплата: 200 руб.
- Просрочка меньше 30 дней → месячная доплата не применяется.
- **Итого: 500 + 200 = 700 руб.**

### Валидация

Система не позволяет:
- Передать отрицательное число дней.
- Передать неизвестный тип книги.

Если дней просрочки 0 — штраф 0.

## 2. Три способа улучшить код

### Улучшение 1: Система лояльности читателей (расширяемость бизнес-логики)

**Проблема:** Сейчас все читатели платят одинаково. В реальной библиотеке студенты, пенсионеры, постоянные читатели и премиум-подписчики заслуживают разных условий.

**Решение:** Ввести категории читателей с коэффициентами скидки.

```python
class ReaderCategory(str, Enum):
    REGULAR = "regular"
    STUDENT = "student"
    SENIOR = "senior"
    PREMIUM = "premium"

DISCOUNT_RATE: dict[ReaderCategory, float] = {
    ReaderCategory.REGULAR: 1.0,
    ReaderCategory.STUDENT: 0.5,
    ReaderCategory.SENIOR: 0.7,
    ReaderCategory.PREMIUM: 0.0,
}
```

**Почему это полезно:**
- Привлекает новых читателей (студенты, пенсионеры).
- Стимулирует оформлять премиум-подписку.
- Легко добавлять новые категории без изменения ядра расчета.

---

### Улучшение 2: Интеграция с внешними сервисами уведомлений (производительность и UX)

**Проблема:** Сейчас система только считает штраф, но не предупреждает читателя. Он узнает о задолженности, только когда придет в библиотеку.

**Решение:** Добавить асинхронную отправку уведомлений по событиям:

| Событие | Действие |
|---------|----------|
| За 1 день до срока возврата | Email/SMS: "Книгу нужно вернуть завтра" |
| Просрочка 1 день | Push: "Вы просрочили книгу, штраф: X руб." |
| Просрочка 7 дней | Email: "Доплата увеличена, верните книгу" |
| Просрочка 30 дней | SMS + блокировка карты читателя |

**Техническая реализация:**
- Использовать `BackgroundTasks` из FastAPI для неблокирующей отправки.
- Или интегрировать Celery + Redis для масштабируемой очереди сообщений.

**Почему это полезно:**
- Снижает количество просрочек и потери книг.
- Улучшает опыт читателя.
- Разгружает библиотекарей от ручных звонков.

---

### Улучшение 3: Конфигурируемая политика штрафов через БД (расширяемость)

**Проблема:** Тарифы, пороги и доплаты захардкожены в Python-коде. Чтобы изменить ставку, нужно деплоить новую версию приложения.

**Решение:** Вынести правила расчета штрафов в таблицу базы данных:

```sql
CREATE TABLE penalty_rules (
    id SERIAL PRIMARY KEY,
    book_type VARCHAR(20) NOT NULL,
    daily_rate INTEGER NOT NULL,
    weekly_surcharge INTEGER NOT NULL,
    monthly_surcharge INTEGER NOT NULL,
    week_threshold_days INTEGER DEFAULT 7,
    month_threshold_days INTEGER DEFAULT 30,
    is_active BOOLEAN DEFAULT TRUE,
    valid_from DATE NOT NULL,
    valid_to DATE
);
```

**Как это работает:**
- При старте приложение загружает активные правила из БД в кэш.
- Администратор библиотеки меняет ставки через админ-панель.
- Можно задавать сезонные акции (например, летом штрафы ниже).
- Можно хранить историю изменений для аудита.

**Почему это полезно:**
- Библиотекарь сам управляет ставками без разработчика.
- Быстрый откат к старым правилам при ошибке.
- Возможность A/B-тестирования разных политик штрафов.

## 3. Итог

Текущий модуль штрафов — простой, понятный и хорошо протестирован. Он решает базовую задачу, но для продакшена стоит добавить:

1. **Скидки по категориям читателей** — для маркетинга и лояльности.
2. **Уведомления** — для снижения просрочек и улучшения UX.
3. **Конфигурация через БД** — для гибкого управления без деплоя.

Эти улучшения не сломают существующую логику, но сделают систему гораздо более гибкой и удобной для бизнеса.
